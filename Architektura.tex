\chapter{Návrh architektúry systému}
\label{NavrhArchitektury}

Táto kapitola popisuje celkovú architektúru navrhovaného riadiaceho systému. Vychádza z~požiadaviek definovaných v~predchádzajúcej kapitole a~z~analýzy dostupných technológií. Popisuje hardvérovú topológiu, softvérovú štruktúru, komunikačnú vrstvu a~formát konfiguračných súborov.

% ------------------------------------------------------------------
\section{Požiadavky na systém}
\label{PoziadavkyNaSystem}

Táto kapitola definuje hranice projektu a podmienky, ktoré musí výsledný systém spĺňať.

\begin{itemize}
    \item \textbf{Funkčné požiadavky:}
    \begin{itemize}
        \item Bezdrôtové riadenie výkonových prvkov (jednosmerné motory s ovládaním rýchlosti/smeru).
        \item Spínanie sieťového napätia 230\,V (osvetlenie, dymostroj).
        \item Natívne a synchronizované prehrávanie multimédií (audio a video).
        \item \textbf{Interaktivita:} Schopnosť asynchrónne vetviť dej na základe hardvérových vstupov (tlačidlá, senzory).
        \item Podpora časovania udalostí pomocou časovej osi (\textit{timeline}).
    \end{itemize}
    \item \textbf{Nefunkčné požiadavky:}
    \begin{itemize}
        \item \textbf{Modularita:} Dizajn umožňujúci jednoduché pridávanie nových uzlov do existujúceho ekosystému.
        \item \textbf{Softvérové oddelenie:} Zmena scenára expozície výhradne úpravou textových konfiguračných súborov (bez zásahu do C++ alebo Python kódu).
        \item \textbf{Dostupnosť:} Riešenie založené na bežne dostupných a overených "off-the-shelf" komponentoch namiesto drahých priemyselných PLC.
    \end{itemize}
\end{itemize}

\section{Celková architektúra a koncepcia}
\label{CelkovaArchitektura}

Systém je navrhnutý na základe distribuovanej IoT architektúry, ktorá oddeľuje výpočtovú logiku od výkonných jednotiek.

\begin{itemize}
    \item \textbf{Topológia hviezdy (Star topology):} Centrála a uzly nie sú spojené pevnou káblovou zbernicou, ale komunikujú bezdrôtovo cez lokálnu sieť Wi-Fi.
    \item \textbf{Centrálna výpočtová jednotka (SBC):} 
    \begin{itemize}
        \item Slúži ako "mozog" systému.
        \item Sleduje stavy, koordinuje \textit{timeline} a priamo prehráva graficky a dátovo náročné multimédiá.
    \end{itemize}
    \item \textbf{Distribuované periférie (Mikrokontroléry):} 
    \begin{itemize}
        \item Slúžia ako "svaly" a zmysly systému.
        \item Vykonávajú lokálnu obsluhu senzorov a riadia špecifické aktuátory (motory a reléové výstupy).
    \end{itemize}
\end{itemize}

\section{Hardvérová topológia}
\label{HardverovaTopoligia}

Hardvérové rozvrhnutie miestnosti s dôrazom na minimalizáciu centrálnej kabeláže.

\begin{itemize}
    \item \textbf{Raspberry Pi ako centrálny uzol:} Fyzicky oddelený od expozície, pripojený priamo k audio/video technike (projektory či displeje cez HDMI, zvukové zosilňovače).
    \item \textbf{Sieť ESP32 modulov:} Implementácia dedikovaných bezdrôtových uzlov priamo pri interaktívnych objektoch:
    \begin{itemize}
        \item \textbf{Senzorický uzol} (Trigger Button) pre detekciu vstupov.
        \item \textbf{Motorický uzol} (\texttt{controller\_MOTORS}) pre fyzický pohyb.
        \item \textbf{Reléový uzol} (\texttt{controller\_RELAY}) pre spínanie efektov 12V a osvetlenia či iných zariadení pracujúcich na 230\,V.
    \end{itemize}
\end{itemize}

\section{Princíp softvérového jadra (Scene Engine)}
\label{SoftverovArchitektura}

Namiesto rigidného sekvenčného (zhora-nadol) programovania využíva hlavná riadiaca logika asynchrónne riadenie stavov.

\begin{itemize}
    \item \textbf{Konečný stavový automat (FSM):} Priebeh expozície (scény) je rozdelený na samostatné, logicky ohraničené stavy.
    \item \textbf{Životný cyklus stavu:}
    \begin{itemize}
        \item \textbf{Inicializácia (\texttt{onEnter}):} Okamžité jednorazové akcie po vstupe do stavu (napr. zapnutie ambientného svetla).
        \item \textbf{Časová os (\texttt{timeline}):} Asynchrónne vyvolávanie akcií s definovanými sekundovými oneskoreniami.
        \item \textbf{Pozorovací režim (\texttt{transitions}):} Systém paralelne čaká na vonkajšie udalosti z prostredia, nazývané prechody (tzv. \texttt{timeout}, \texttt{audioEnd}, \texttt{videoEnd}, \texttt{mqttMessage}).
    \end{itemize}
    \item \textbf{Dynamické prechody:} Splnením prechodovej podmienky sa aktuálny stav preruší a automat sa presunie do nového cieľového stavu, čím sa dosahuje požadované interaktívne vetvenie výstavy.
\end{itemize}

\section{Architektúra komunikačnej vrstvy}
\label{KomunikacnaVrstva}

Výmena dát medzi stavovým automatom (Python) a perifériami zavádza asynchrónne štandardy a obmedzuje tvrdokódované postupy.

\begin{itemize}
    \item \textbf{Protokol MQTT:} Komunikácia na princípe \textit{Publish/Subscribe} cez konštantné pripojenie na TCP/IP broker rozmiestnený nad doménou.
    \item \textbf{Štruktúra zverejňovania:} 
    \begin{itemize}
        \item ESP32 neustále odoberajú svoj priradený topik (napríklad \texttt{room1/motor1}).
        \item Pri príchode do bloku \textit{scene engine} odošle Raspberry príkaz zverejnením správy (payloadu) a prepne sa do režimu čakania na spätnú väzbu na dedikovanom topiku s príponou \texttt{/feedback}.
    \end{itemize}
    \item \textbf{LWT (Last Will and Testament):} Detekcia \textit{online/offline} stavu všetkých komponentov riešená neustálym udržiavaním zostatkovej (retained) správy s automatickým prepisom pri strate spojenia s uzlom.
    \item \textbf{Separácia dát z kódu formátom JSON:} Všetka choreografia bola vytiahnutá z aplikačnej logiky do externých súborov \texttt{.json}. Editorom tohto textového kontraktu určujúceho \textit{timeline} či \textit{onEnter} prechody je výhradne Game Master, bez potreby rekompilácie aplikačného balíčka.
\end{itemize}
